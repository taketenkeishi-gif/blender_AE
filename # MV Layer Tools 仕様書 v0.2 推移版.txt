# MV Layer Tools 仕様書 v0.2 推移版

## 0. この文書の目的

この文書は、既存の「Blender AE風環境 3本柱 同時進行 企画書」を、実装検証の結果に合わせて更新していくための推移版仕様書である。

初期企画では、Blender Pythonアドオンとして、静止画MV・2D演出・画像合成向けのAE/AviUtl風制作環境を作ることを目標にしていた。現時点では、NパネルUI、C1外部JPEGプレビュー、Blender本物Viewport直接操作の検証を経て、設計上の重要な分岐が発生している。

この文書では、初期方針を破棄せず、検証結果を反映しながら、次の実装対象を具体化する。

---

## 1. 現在の結論

### 1.1 採用する大方針

現時点の最重要結論は以下である。

```text
プレビュー画面・直接ドラッグ編集:
  Blender本物Viewportを使う

AviUtl風タイムライン・レイヤー一覧・プロパティ編集:
  Blender内補助UI、または外部UIとして別途設計する

確認用サムネイル・低頻度プレビュー:
  C1 JPEG preview方式を補助的に使う余地はある

最終レンダー・エンコード前レンダー:
  Blender Render / Eevee / Cycles / Compositor を使う
```

### 1.2 不採用にする方針

以下は、現時点では本線にしない。

```text
外部UIにJPEG/PNGプレビューを表示し、
その上で直接ドラッグ編集する方式
```

理由は、C1検証で以下が確認されたためである。

* JPEGプレビュー表示そのものは成立した
* しかし直接ドラッグ編集では遅延が大きい
* UI側操作枠とBlender側オブジェクト位置にズレが出た
* state更新とpreview更新のタイミング差でワープが発生した
* 操作感が実用レベルに達しなかった

一方で、Blender本物Viewport上でPlaneを直接ドラッグするMVPでは、以下が確認された。

* 操作枠とPlaneがズレない
* ワープしない
* JPEG/HTTP/command/state経由の遅延がない
* 同じ用途で比較すると、C1方式より明確に快適

したがって、直接操作の中核はBlender Viewportに置く。

## 2. 初期企画から維持する3本柱

初期企画の3本柱は維持する。ただし、それぞれの実装場所と責務を再整理する。

### 2.1 柱1: AE / AviUtl風タイムライン

目的は、Blender標準のDope SheetやTimelineを根本置換することではない。

2D素材を時間軸で扱うための操作層を作る。

最低限必要な要素は以下。

* レイヤーごとの開始フレーム
* レイヤーごとの終了フレーム
* 表示区間の編集
* キーフレーム対象の明確化
* Position / Rotation / Scale / Opacity のキー追加
* 2D / 擬似3Dレイヤーの時間範囲表示
* 将来的なバーUIへの拡張余地

現時点の追加方針:

```text
タイムラインUIは、Blender Viewportとは分離して設計する。
Viewportはプレビューと直接操作を担当し、
タイムラインは時間範囲・キー・レイヤー順を担当する。
```

### 2.2 柱2: 素材取り込み・レイヤー管理

素材は単なるPlaneではなく、MV用レイヤーとして扱う。

最低限必要な要素は以下。

* 複数画像読み込み
* 連番画像読み込み
* 画像Plane化
* 透過対応Material生成
* 専用Collection整理
* レイヤー順序管理
* 表示 / 非表示
* ロック
* 選択同期
* グループ / カテゴリ
* source_path保持
* 将来PSDや差分管理へ伸ばせる情報保持

現時点の追加方針:

```text
レイヤーの選択・直接操作はViewport上でも行う。
レイヤー一覧UIは、選択状態・表示状態・時間範囲・カテゴリを管理する。
```

### 2.3 柱3: 平面素材向け演出UI

Blender標準Transform UIやノード画面に潜らず、2D MV頻出演出を素早く操作する。

最低限必要な要素は以下。

* 位置
* 回転
* 拡大縮小
* 不透明度
* Fade In / Out
* Shake
* Zoom
* 2Dカメラ操作
* 擬似3D操作
* 将来的なクリッピング風・ゆらめき風・色味補正

現時点の追加方針:

```text
頻繁な直接操作はViewport modal operatorで行う。
数値編集・プリセット適用・キーフレーム化はPanel/Timeline/Property UIで行う。
```

## 3. プレビュー方式の検証結果

### 3.1 C1: 外部JPEG/PNGプレビュー方式

C1方式は以下の構成で検証した。

```text
Blender
  ↓ OpenGL render
preview.jpg / preview.png
  ↓ HTTP / file read
外部UI
  ↓ command.json
Blender
```

### 3.2 C1ベンチ結果

C1 split benchmarkでは、以下の傾向が確認された。

```text
Scene update only:
  十分高速

OpenGL + PNG write:
  15fps前後

OpenGL + JPEG write:
  30fps前後
```

この結果から、単なる確認用JPEGプレビューとしては成立する可能性がある。

### 3.3 C1 Drag MVP結果

C1 Drag MVP v1/v2/v3では、以下が確認された。

```text
v1:
  一応追従するが遅延が大きい

v2:
  操作枠はある程度許容できる
  Blenderプレビューはひたすらラグい

v3:
  反応開始は早い
  しかしオブジェクトがワープする
  操作枠とドラッグのリンクも不安定
```

### 3.4 C1の結論

C1は以下の用途には使える可能性がある。

* 外部UIの確認用サムネイル
* タイムライン上の低頻度プレビュー
* 最終レンダー前の簡易確認
* バックグラウンドプレビュー生成

C1は以下の用途には使わない。

* メインプレビュー画面
* 直接ドラッグ編集
* 正規の座標判定
* Viewport代替

## 4. Blender本物Viewport直接操作方式

### 4.1 採用理由

Viewport直接操作MVPでは、以下の流れで検証した。

```text
Blender Viewport上のマウス操作
  ↓
modal operator
  ↓
Plane.locationを直接更新
  ↓
Blender GPU Viewportが即時描画
```

この方式では、C1で問題になった以下の経路を挟まない。

* JPEG保存
* HTTP取得
* command.json往復
* state.json同期
* 外部UI側の推定座標
* 遅延したscreen_rect

結果として、直接操作用途では最も自然だった。

### 4.2 Viewport直接操作MVPで確認できたこと

確認できた内容:

* PlaneをViewport上で直接ドラッグできる
* 選択枠とPlaneがズレない
* 操作中にワープしない
* 同じ用途ではC1より明確に快適

未確認の内容:

* 複数レイヤー選択
* 重なったレイヤーのクリック判定
* 透過PNG素材での当たり判定
* 回転・拡大縮小ハンドル
* Z移動
* 擬似3D回転
* キーフレーム挿入との統合
* レイヤー一覧UIとの同期
* タイムラインUIとの同期

### 5.1 採用アーキテクチャ

```text
Blender Viewport Layer Editor
  - 実プレビュー
  - 直接ドラッグ
  - 選択枠
  - ハンドル
  - 擬似3D操作
  - カメラ表示
  - GPU Viewport描画

MV Layer Core
  - Layer PropertyGroup
  - Scene PropertyGroup
  - TimelineModel
  - LayerModel
  - EffectModel
  - Service層

Timeline / Layer UI
  - AviUtl風タイムライン
  - レイヤー一覧
  - キーフレーム編集
  - エフェクト一覧
  - 数値プロパティ

Render / Export
  - Blender Render
  - Eevee / Cycles
  - Compositor
  - 最終動画/連番出力

Optional Passive Preview
  - C1 JPEG preview
  - サムネイル
  - 外部UI低頻度確認
```

### 5.2 設計原則

* プレビュー画面は外に出さない
* 直接操作はBlender Viewport上で行う
* タイムラインUIはViewportから分離する
* Blender標準UIを完全置換しない
* Python APIで届く範囲を最大化する
* fork判断は保留し、必要性が観測で確定してから行う
* Operatorは薄く、実処理はserviceへ寄せる
* Viewport操作系は専用moduleへ分離する

## 7. 擬似3Dモード仕様

### 7.1 基本思想

本プロジェクトの「2D」は、最終出力と素材性質が2Dであるという意味である。

編集空間には、AviUtl拡張編集に近い擬似3D概念を持たせる。

```text
素材の本質:
  2D画像 / 動画 / テキスト

編集空間:
  2.5D / 擬似3D

最終出力:
  2D映像
```

### 7.2 レイヤーの最小Transform

各レイヤーは以下を持つ。

```text
position_x
position_y
position_z
rotation_x
rotation_y
rotation_z
scale_x
scale_y
anchor_x
anchor_y
opacity
frame_start
frame_end
```

Blender内部では以下へ対応させる。

```text
position_x -> Object.location.x
position_y -> Object.location.y または camera depth
position_z -> Object.location.z
rotation_x -> Object.rotation_euler.x
rotation_y -> Object.rotation_euler.y
rotation_z -> Object.rotation_euler.z
scale_x -> Object.scale.x
scale_y -> Object.scale.y
opacity -> Material alpha / custom property
```

注意:

現行MVPでは、カメラ正面Planeとして `x / z` を画面平面、`y` を奥行きとして扱っている。今後、UI表記上の `X/Y/Z` とBlender内部の `x/y/z` 対応を仕様として固定する必要がある。

### 7.3 操作モード

Viewport上の最小操作は以下。

```text
左ドラッグ:
  画面平面上の移動

Shift + 左ドラッグ:
  Z / depth移動

Alt + 左ドラッグ:
  回転

Ctrl + 左ドラッグ:
  拡大縮小

Ctrl + Shift + 左ドラッグ:
  縦横比固定スケール

Esc:
  modal操作停止
```

### 7.4 ハンドル仕様

最初に実装するハンドル:

* 中心点
* 外枠
* 四隅スケールハンドル
* 回転ハンドル
* Z移動ハンドル

後回し:

* 3軸回転ギズモ
* 透視投影下の精密操作
* 透明ピクセルクリック判定
* マスク境界編集

## 10. Fork判断ライン

### 10.1 まだforkしない理由

Viewport直接操作MVPが快適だったため、現時点でBlender forkは不要である。

まずは以下で進める。

```text
Python modal operator
Viewport draw handler
PropertyGroup
Service層
Nパネル/補助UI
必要なら外部Timeline UI
```

### 10.2 forkを検討する条件

以下が観測で確定した場合のみforkを検討する。

* Python modal operatorではハンドル操作が不安定
* draw handlerでは必要な描画性能やUI密度に届かない
* Blender標準イベント処理と競合して操作感が改善できない
* 独自タイムラインをBlender内Editorとして統合する必要がある
* Blender Python APIで必要な内部状態へ到達できない
* Undo/Redoや選択同期がPython層では安定しない
